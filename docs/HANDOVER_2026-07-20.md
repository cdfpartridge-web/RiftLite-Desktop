### Atlas OAuth root cause and fix

The repeated Clerk `authorization_invalid` error was not an account-data failure. RiftAtlas Google login currently performs a same-window redirect. RiftLite's older navigation policy sent that redirect to the default browser, while Clerk's required client cookies remained inside `persist:riftlite-atlas`. The browser therefore reached the callback without the matching Clerk state and correctly returned HTTP 403.

The released fix:

- keeps only the exact approved HTTPS Atlas/OAuth origins in the persistent embedded session;
- retains narrow sandboxed popup handling when providers do use a popup;
- blocks unknown origins, ports, and cross-provider callbacks;
- detects the exact Clerk callback failure;
- clears only Clerk authentication cookies for Atlas during recovery, preserving unrelated cookies and all RiftLite data.

## Web Replay Architecture and Privacy

Preserve this flow:

```text
Atlas WebSocket
-> game preload (primary) / debugger fallback
-> central frame deduper
-> keyed RawCaptureService sessions
-> private local raw JSON and index
-> immutable private upload
-> server canonical normalization and privacy projection
-> RiftLite website player / desktop embed
-> optional idempotent Discord report
```

Important invariants:

- Preload is primary; the debugger path is fallback only.
- Do not return to one global raw-capture session. Sessions are keyed and associated through capture, series, match, replay, room, and strict temporal identity.
- A BO3 is one series with explicit child-game boundaries and identities.
- Raw artifacts are processing/diagnostic input. Canonical replay artifacts are playback truth.
- TCGA did not feed Web Replay V2 at this historical checkpoint; the v0.9.10 release candidate adds the isolated TCGA production capture path described in the current handover.
- Hosted player fixes can deploy without rebuilding desktop because the app embeds the website player.
- Website Web Replay currently supports `1x`, `2x`, `4x`, `6x`, and `10x`.
- First-party website processing, private/unlisted/public access, embedded desktop library sessions, owner diagnostics, BO3/sideboard presentation, current card identity mapping, and the consented two-perspective combiner are deployed. The manual combiner remains hidden from the ordinary embedded library.
- Raw source artifacts remain private. Only the canonical privacy-projected replay follows Private, Unlisted, or Public visibility.
- Public/canonical output must not expose tokens, room IDs, raw payloads, or opponent-private state.

## Atlas Turn-by-Turn Logs

The tester's latest question was whether RiftLite saves turn-by-turn Atlas data or only video. The answer is: it saves the underlying action stream as well as optional video, when RiftLite Web Replay/raw capture has been enabled.

Default raw-capture locations:

- Windows: `%USERPROFILE%\Documents\RiftLite\Replay Bundles\Raw Capture`
- macOS: `~/Documents/RiftLite/Replay Bundles/Raw Capture`
- A custom Replay folder is respected.

Each completed capture has:

- `<match-title>-<captureSessionId>.json`
- an adjacent `.riftlite-index.json` manifest

The raw JSON includes ordered/timestamped WebSocket frames, direction, socket, parsed type, exact raw payload, lifecycle phases, game boundaries, and capture identity. It is the source from which deterministic Web Replay actions and frames are reconstructed. It is independent of MP4 recording, so a video failure does not imply the action data was lost.

The normal `.riftreplay` v4 coaching export includes stored capture/structured events such as setup, mulligan, turn start/end, draw, play, move, score, combat, result, battlefield, and generic actions. It deliberately removes the local raw path and does not embed the complete raw sidecar.

Current UI route:

1. Open **Settings -> Capture & replays**.
2. Use **Open folder** in the Replay section.
3. Open the `Raw Capture` subfolder.

Current limitations:

- Raw capture is Atlas-only and opt-in; video-only historical matches cannot be converted into raw action logs.
- There is no dedicated one-click readable/privacy-safe action-log export yet.
- `riftlite-capture-events.jsonl` is rotating support diagnostics, not the authoritative full match action log.
- A capture is capped at 10 MiB or 12,000 messages; ordinary matches are normally well below this.
- Completed raw sidecars currently persist until manually removed; there is no retention UI/garbage collector.
- Raw files can include names, room codes, chat, local hidden information, deck state, and other match data. Never post them publicly or include them in ordinary diagnostics.

Strong next feature: add **Export action log** with privacy-safe JSON and CSV derived from canonical/structured events. Suggested columns are game, turn/time, actor, action, card, source/destination, battlefield, points, and score. Exclude chat, room codes, tokens, raw payloads, internal IDs, and opponent-private information by default. Cover BO3 boundaries, undo/unknown events, alternate-art normalization, and privacy projection in tests.

## Local RiftAtlas Battlefield-Ownership Hotfix (Not Published)

Tester diagnostics `riftlite-capture-events (53).jsonl` contain 242 valid records covering two Atlas BO3 sessions and no TCGA session. The newest unswapped 2-1 draft captured these battlefield pairs:

- game 1: local `Frozen Fortress`, opponent `Treasure Hoard`;
- game 2: local `Void Gate`, opponent `Vilemaw's Lair`;
- game 3: local `Sigil of the Storm`, opponent `Sunken Temple`.

The tester reported that every Atlas game that day was inverted and that this newest match had not been manually corrected. The corrected newest pairs are therefore `Treasure Hoard` / `Frozen Fortress`, `Vilemaw's Lair` / `Void Gate`, and `Sunken Temple` / `Sigil of the Storm` for local/opponent respectively.

Root cause: the desktop globally assumed `battlefieldB` was local and `battlefieldA` was the opponent. Atlas ownership is seat-dependent: battlefield A belongs to seat 0 and battlefield B belongs to seat 1. A global A/B swap would fix seat 0 while regressing seat 1.

The local fix now:

- reads the authoritative numeric seat from inbound Atlas `room_shell_sync` (`sessionDoc.selfPlayer`) and `authoritative_snapshot` (`snapshot.players`) frames;
- identifies the local record only by exact match with the WebSocket URL `playerId`, never player-array order, display name, battlefield content, or who went first;
- maps A/B per seat, resets the cached seat when the BO3 advances to a new room, and ignores stale callbacks from an older socket;
- fails closed with blank battlefield fields until seat evidence arrives instead of guessing an owner;
- includes privacy-safe `atlasLocalPlayerSeat`, local zone, and opponent zone fields in future support diagnostics without retaining the private player ID;
- leaves replay-viewer lane orientation unchanged because the current replay frame model does not yet carry authoritative local seat; a separate engine-to-viewer seat propagation is required before that can be changed safely.

## Security and Privacy State

- Account refresh tokens and integration secrets use Electron `safeStorage` (Windows DPAPI or macOS Keychain-backed storage).
- New manual and cloud backups omit secrets.
- Default diagnostic bundles redact names, handles, email addresses, room codes, tokens, URLs/queries, local paths, and raw payloads.
- A separate sensitive export requires explicit explanation and confirmation.
- App IPC validates the exact trusted renderer origin.
- Game IPC validates provider, partition, schema, payload size/depth, rate, and active session.
- Atlas, TCGA, and replay webviews use narrow origin, navigation, popup, permission, and download policies.
- An anonymous Firebase credential is infrastructure authentication, not evidence that the desktop is linked to a RiftLite account.
- Reconnect is pinned to the stored account UID. Switching accounts is an explicit unlink/relink operation.
- Cloud backup chunks are immutable per generation; the manifest changes only after checksum verification.
- Old retained SQLite recovery `.bak` files are not automatically rewritten. Pre-hardening backups can retain plaintext secrets until normal ten-backup rotation replaces them or the user deliberately removes them.
