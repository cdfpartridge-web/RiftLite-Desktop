## What v0.8.04 Changed

### Private Hub Co-owners

- Hub owners can promote a trusted member to co-owner.
- Co-owners can perform supported member, invite, moderation, testing-goal, and Discord administration actions.
- The original owner remains distinct and in control of owner-only actions.
- Website and desktop permission checks now share the same capability model instead of relying on scattered owner/admin comparisons.
- The central website permission logic is in `src/lib/social/hub-permissions.ts` and `src/lib/social/server.ts`.

### Hub Health

- Private hubs now expose a health view that brings together:
  - the exact RiftLite hub ID
  - linked account/profile state
  - Discord server configuration
  - member verification state
  - latest web replay
  - replay upload and Discord delivery stages
- Website endpoint: `src/app/api/hubs/[hubId]/health/route.ts`.
- Website UI: `src/components/site/my-hubs-client.tsx`.
- This is the first diagnostic surface to inspect when a hub says verification, account identity, replay upload, or Discord reporting is not working.

### Discord `/verified`

- `/verified` lists members in the connected server who have linked RiftLite accounts.
- It uses current RiftLite display names and handles.
- It deliberately does not expose email addresses or internal Firebase UIDs.
- Discord administrators can use it to distinguish a genuinely linked user from a stale Discord role or an unlinked hub member.
- Command registration: `scripts/register-discord-commands.mjs`.
- Handler: `src/app/api/discord/bot/interactions/route.ts`.
- Formatting: `src/lib/discord/verified-members.ts`.

### Reliable Automatic Replay Reporting

The FlashShad0w investigation established that a valid replay could upload successfully but never create a Discord share claim. The desktop had snapshotted replay-sharing consent on an early Atlas lobby/prelude WebSocket frame, before the real match started. If the user opted in or changed hubs after that early frame, the completed match could silently miss Discord sharing.

The durable fix is in `src/main/services/rawCaptureService.ts`:

- Account and sharing consent are refreshed at the first authoritative Atlas match frame, such as `room_shell_sync` or equivalent strong match identity.
- The replay may be shared only to hubs selected at the real match start and still selected when the match completes.
- Opting out before completion prevents the share.
- Retry eligibility keeps intended hubs recoverable without sharing to a hub that is no longer selected.
- The final persisted match result is awaited before the first Discord-eligible upload.
- The existing result wait is 15 seconds followed by polling every 2.5 seconds for up to 30 seconds.
- Resolved result summaries are not replaced by later incomplete evidence.
- Capture, result, upload, website processing, Discord claim, and Discord message stages are persisted so interrupted work can retry without duplicating a successful post.

Regression tests were added to `tests/rawCaptureService.test.ts`, including:

- opting in during the prelude but before the authoritative match start
- sharing only the match-start/completion hub intersection
- retry and completed-stage behaviour

### Account And Handle Continuity

The account system must remain additive and backwards compatible:

- Do not force existing users to create a new account or re-sign.
- A desktop anonymous Firebase identity may remain part of the durable identity graph after Google/email linking.
- Website and desktop identities are associated server-side; a linked provider account is not assumed to replace every historical UID.
- Memberships, ownership, replay indexes, inbox entries, Discord links, identity snapshots, and cloud backup references are migrated or associated without deleting source records.
- Existing canonical cloud backups are never silently replaced during account association.
- Current server profile identity supplies hub names and Discord report names, preventing random Firebase UID-like handles from remaining visible after verification.
- Replay ownership accepts verified durable identities rather than relying only on the current `sign_in_provider` label.

See `docs/account-onboarding.md` before changing any account-link or migration behaviour.

## Discord Production Configuration

During the v0.8.04 publish, the live environment was found to contain mismatched Discord application values:

- the community bot token was valid
- the bot's configured interaction endpoint already pointed at `https://www.riftlite.com/api/discord/bot/interactions`
- `DISCORD_APPLICATION_ID`, `DISCORD_PUBLIC_KEY`, and `DISCORD_GUILD_ID` did not match the application/server represented by that token
- the bot itself was installed in `Team UK - Riftbound`

The three community-bot values were corrected in Vercel production, the website was redeployed, and the commands were registered against the bot's actual server.

Current registered commands:

- `/leaderboard`
- `/recent`
- `/setup`
- `/testing-goals`
- `/verified`
- `/verify`
- `/weekly-report`

Important distinction:

- `DISCORD_APPLICATION_ID`, `DISCORD_PUBLIC_KEY`, `DISCORD_GUILD_ID`, and `DISCORD_COMMUNITY_BOT_TOKEN` belong to the community/testing bot.
- `DISCORD_CLIENT_ID` is also used by the separate Discord RPC OAuth route. It was intentionally not overwritten merely because it differs from the community bot application.
- Bot tokens and client secrets must never be printed, committed, sent to the desktop, stored in readable Firestore documents, or included in diagnostics.

The production interaction route rejects unsigned requests. A safe smoke test is to confirm that an unsigned POST returns `401`; do not send a real Discord message merely to test deployment.

## Web Replay System

The complete replay architecture is documented in `docs/WEB_REPLAY_SYSTEM_HANDOVER.md`. Treat that document as required reading rather than attempting to infer Replay V2 from the UI.

Key invariants:

- Atlas raw capture and automatic website upload are explicit, account-bound opt-ins.
- Capture and public/external upload consent remain separate.
- Automatic uploads are private/unlisted by default according to the current product flow.
- An unlisted replay is viewable by anyone with its link but is absent from the public library.
- Replay artifacts are immutable; fixes normally create a new projection/generation rather than rewriting historical raw evidence.
- BO3 is one series with child games, between-game sideboarding, per-game battlefield/setup/mulligan state, and a final series score.
- The desktop embed uses a short-lived secure HttpOnly website session. Upload/mutation APIs remain bearer-token protected.
- Combined two-perspective replays default to unlisted.
- A **Hidden** badge is valid only when Atlas explicitly sets `hidden: true` on a card physically at a battlefield and it has not been revealed to the opponent.
- Do not infer hidden status merely because a card came from only one player's hand/deck perspective.
- Automatic Discord replay sharing posts the unlisted link to the configured reports channel after the final result is available.

## Known Caveats And Guardrails

- The desktop renderer is still concentrated in the very large `src/renderer/App.tsx`; avoid broad formatting churn.
- Atlas WebSocket frames can arrive through preload and debugger paths; retain the central frame deduper and debugger fallback-only behaviour.
- Raw captures are keyed sessions. Do not return to one global active capture.
- BO3 dedupe must use explicit session/game/result identity, not matching scores alone.
- Cloud restore and account sync hardening from July 9 must remain intact.
- The website branch contains a large accumulated feature set. Review exact diffs before rebasing or merging the draft PR.
- Do not expose the old local Replay Lab or Vision features unless explicitly requested.
- Do not treat the account's current provider label as the complete durable identity.
- Never use an early Atlas lobby frame as match-start consent.
- Never post a Discord replay solely because upload finished; use the persisted delivery stages and finalized score.
- Do not change replay visibility from unlisted/private to public without an explicit user action.
