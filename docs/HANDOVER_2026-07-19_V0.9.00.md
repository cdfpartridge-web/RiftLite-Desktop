## Account Confidence, Security, and Privacy

- Account Sync shows local/cloud match and deck counts, last successful sync, active-deck state, data-source/device time, and migration progress.
- Choosing local or cloud data presents a preview before applying the choice.
- Account and two-device conflict tests are mandatory release-gate checks.
- Refresh tokens and integration secrets migrate to Electron secure storage: Windows DPAPI or macOS Keychain-backed storage.
- Newly created manual and cloud backups omit secrets by default.
- Diagnostic bundles redact names, handles, email addresses, room codes, tokens, raw payloads, and local paths unless a separate sensitive export is explicitly confirmed.
- App IPC validates the trusted renderer origin. Embedded Atlas and TCGA guests use narrow provider, partition, navigation, popup, permission, and payload policies.

Existing retained SQLite recovery `.bak` files are not automatically rewritten or deleted. A backup created before this hardening can retain an old plaintext value until normal ten-backup rotation replaces it or the user deliberately removes it. Newly created manual and cloud backups are sanitized.

## Atlas Embedded Sign-in Hotfix

On 2026-07-19 the rebuilt v0.9.00 candidate exposed a Clerk `authorization_invalid` response after starting RiftAtlas sign-in inside RiftLite. The final installed retry was correlated against the app capture timestamps and a read-only copy of Opera GX history. At the exact failure time, `accounts.google.com` and `https://clerk.riftatlas.com/v1/oauth_callback` ran in Opera rather than RiftLite. The temporary history copy was then removed; the real browser profile was not changed.

- RiftAtlas stores Clerk's required client state in cookies under `persist:riftlite-atlas`; the provider callback must run in that same Electron session.
- The root cause was a split navigation policy. Approved OAuth origins were retained for `window.open` popups, but Clerk currently performs Google sign-in as a same-window redirect. RiftLite's main-frame guard blocked that redirect and explicitly handed it to the default browser. Opera did not have the Clerk client state from the embedded Atlas partition, so Clerk correctly returned HTTP 403 `authorization_invalid`.
- `isAllowedGameMainFrameNavigation` now keeps only the exact approved default-port HTTPS game/OAuth origins inside the persistent game session. Unknown origins, non-default ports, cross-provider callbacks and `about:blank` remain blocked. Unrelated external links retain the normal external-browser behavior.
- Popup OAuth continues to share the exact parent session with sandboxing, context isolation, disabled Node integration, web security and the narrow provider allowlist.
- Exact Clerk callback failure recovery now covers both popup and same-window flows. It detects only HTTP 403 at the exact Clerk `/v1/oauth_callback` path, with the exact Clerk JSON error parser retained as a fallback.
- Recovery clears only Atlas Clerk authentication cookies (`__client*`, `__session*`, `__refresh*`, and `__clerk*`), preserving Cloudflare and unrelated Atlas cookies. RiftLite account data, matches, decks and replays are untouched.
- Focused popup/content-security coverage passed 13 tests. The final complete release gate passed 68 files / 552 tests, including the separate 49-test account/two-device gate, TypeScript, production builds, and NSIS packaging.
- Final local installer: 200,612,562 bytes; SHA-256 `4E1FF08E4630D86646526B59ECF54EC91DBB31750AB4F2E770CDD4CC5F63C97A`. `latest.yml` reports `0.9.0`, and its size and SHA-512 match the installer. NSIS archive integrity passed.
- No commit, GitHub push, tag, release, or installer upload was made.

## Atlas Local Video and Replay-State Hotfix

The final release candidate also fixes the contradictory state reported by replay `rl2_41f5854faf062d7828dcd00b9c104fa1`: the server replay was Ready and Discord had posted it, while the open desktop detail panel continued to show an earlier pre-upload snapshot.

- `RawCaptureService` now emits each saved replay transition to the trusted renderer, and the renderer replaces the existing list record without changing its order.
- Delivery copy distinguishes Preparing, Queued, Uploaded, Ready, and Shared states. A ready Web Replay is no longer labelled Missing media merely because optional local video was unavailable.
